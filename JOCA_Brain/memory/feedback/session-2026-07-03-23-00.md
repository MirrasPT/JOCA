**Categoria:** workflow-gap
**Severidade:** medium
**Descrição:** O user prefixou uma mensagem com "Apontas aí para fazer:" seguida de 3 itens de correcção, marcando explicitamente só 1 deles como "não é fazer já, aponta aí só para fazer depois". O main loop interpretou os outros 2 itens como GO imediato por conterem frases de elaboração com verbo imperativo (ex.: "Faz as edições necessárias para o back-end e o frontend..."), e avançou para deploy ao vivo (vídeos + imagens + código) num site pessoal do user antes dele poder rever. O user corrigiu: o prefixo "aponta aí para fazer" cobre a lista inteira por omissão — uma frase de elaboração dentro de um item descreve o que a correcção implicaria, não é necessariamente ordem de executar já.
**Componente afectado:** `rules/task-intake.md` (secção Segurança / classificação de vias) não tem regra explícita para este padrão de linguagem ("nota para depois" vs "GO imediato") quando misturado com frases imperativas dentro da mesma mensagem.
**Fix sugerido:** Adicionar a `rules/task-intake.md` uma heurística: se a mensagem abre com linguagem de "apontar/notar/pendente/para fazer depois", tratar TODOS os itens da lista como backlog por omissão — só executar de imediato um item se houver sinal de GO isolado e inequívoco para esse item específico (não apenas uma frase que descreve o trabalho envolvido). Em caso de ambiguidade real, perguntar antes de agir, sobretudo se a acção é visível/ao vivo (deploy, publicação, envio). Memória de utilizador já criada (`renato-note-vs-do-now.md`) cobre isto a nível de perfil pessoal — mas o gap estrutural em `task-intake.md` beneficiaria todos os projectos/users, não só este.
