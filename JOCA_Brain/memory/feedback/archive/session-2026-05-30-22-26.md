# Feedback JOCA — sessão Bracaris (catálogo + copy via Wix REST API)

**Categoria:** missing-skill / skill-improvement | **Severidade:** high | **Componente afectado:** `.claude/skills/wix-cli.md`
**Descrição:** A skill `wix-cli` só cobre projectos Wix CLI (apps + headless sites). Não cobre o caso **muito comum** de um **site Wix Editor clássico já existente** onde o conteúdo do **catálogo Wix Stores se edita via REST API** (não via CLI, que não clona sites Editor). Esta sessão gastou esforço considerável a descobrir do zero: (1) Editor clássico ≠ headless → CLI não aplica; (2) catálogo editável por REST API mesmo assim; (3) CATALOG_V1 vs V3 (V3 dá 428); (4) regra de headers `wix-site-id` **OU** `wix-account-id` nunca os dois ("No Metasite Context"); (5) endpoints query/PATCH de products com `description` HTML, `seoData`, `ribbon`, `brand`.
**Fix sugerido:** Adicionar à skill `wix-cli` (ou criar skill `wix-rest-api`/`wix-stores`) uma secção "Editar site Wix existente (Editor clássico)": detecção Editor vs Studio vs headless (via `<meta generator>` no HTML live); tabela de endpoints Wix Stores V1; regra dos headers metasite; catalog versioning V1/V3; como obter site ID (site-list API ou URL `/dashboard/<id>`). A skill actual roteia para "Velo/Wix CLI" o que induziu em erro a decisão inicial do projecto.

**Categoria:** doc-gap | **Severidade:** medium | **Componente afectado:** `.claude/skills/wix-cli.md` (Project detection / Escalation)
**Descrição:** A tabela de detecção da skill assume sempre projecto CLI local. Não há ramo para "site já publicado, sem projecto local" — que é o ponto de partida de muitos clientes de design.
**Fix sugerido:** Adicionar ao "Escalation"/routing: se não há `wix.config.json` e o objectivo é editar conteúdo de um site existente → inspeccionar o site live (`<meta name="generator">`), classificar (Editor clássico → browser + REST API p/ catálogo; Studio/headless → CLI possível), e só então escolher caminho.

**Categoria:** tool-reliability | **Severidade:** medium | **Componente afectado:** Wix REST API `ecom/v1/discount-rules` (externo)
**Descrição:** A API de discount-rules devolveu dados inconsistentes/efémeros neste site (contagem 17→5→2→0 entre chamadas; regras presentes na query davam 404 no GET; sem endpoint `/disable`). Causa provável: descontos geridos por app 3rd-party que sincroniza em tempo real. Tornou impossível "remover descontos" de forma fiável por API.
**Fix sugerido:** Documentar na skill Wix que descontos criados por apps de terceiros não são geríveis de forma fiável pela REST API nativa — remover/configurar na app ou no dashboard. Não escrever em discount-rules que pareçam app-managed (campo `offer` = "Created by <app>").
