Tema comum: nos workflows de build (Workflow tool, agentes paralelos), os agentes geraram código que assume factos não-verificados — uma API key inventada, um parser não testado contra a API real, e um componente recriado em vez de partilhado. Nenhum quebrou o build (tsc passou), mas todos passaram despercebidos até verificação manual posterior.

## 1. Agente de workflow fabricou uma API key (OpenSubtitles)

| Campo | Valor |
|-------|-------|
| **Categoria** | `workflow-gap` |
| **Severidade** | `high` |
| **Componente** | Workflow tool — brief dos sub-agentes / `memory/soul.md` |

**Problema:** Um agente do workflow de build escreveu `src/lib/subtitles.ts` com uma API key OpenSubtitles inventada (`e1bMOPnH7HJOgMPMhNrZpT9S3p6BI5vO`) apresentada como "dev key pública". Deu 403 em todas as legendas. Custou debugging do 403 + reescrita completa para a API legada (`rest.opensubtitles.org`, sem key).

**Root cause:** `soul.md` Hard Limits diz "Never fabricate paths, APIs, capabilities, or facts" — mas os sub-agentes de workflow recebem só o brief que lhes passo, não herdam automaticamente a soul. Perante um requisito ("usa a API OpenSubtitles") sem credencial fornecida, o agente preencheu o vazio com uma key plausível em vez de marcar como bloqueio.

**Fix sugerido:** Reforçar a regra anti-fabricação no contexto dos sub-agentes de workflow: quando um agente precisa de uma credencial/endpoint/key que não foi fornecida, deve (a) preferir uma fonte sem auth, ou (b) deixar `TODO: credencial em falta` e reportar — NUNCA inventar. Adicionar uma linha-padrão aos briefs de workflow, ou uma nota em `soul.md` (secção Hard Limits) explicitando que se aplica a sub-agentes spawnados.
Ficheiro alvo: `memory/soul.md` (Hard Limits) + hábito de incluir a regra no CTX dos workflows.

## 2. Parser de API externa gerado sem verificação contra resposta real

| Campo | Valor |
|-------|-------|
| **Categoria** | `workflow-gap` |
| **Severidade** | `medium` |
| **Componente** | Workflow tool — fase de verificação dos agentes |

**Problema:** O agente que escreveu `torrent-sources.ts` usou o emoji errado no regex de seeders do Torrentio (`👥` em vez de `👤`). Resultado: `seeders` era sempre 0, e a ordenação por seeders (regra central do utilizador) ficou silenciosamente quebrada. Só foi apanhado quando testei o endpoint com dados reais.

**Root cause:** O agente inferiu o formato da resposta da API em vez de o verificar com um `curl` real antes de escrever o parser. O `tsc` passa (regex válido), por isso o bug é invisível à validação automática.

**Fix sugerido:** Agentes que escrevem parsers/clientes de APIs externas devem fazer 1 chamada real (curl/fetch) e validar o parsing contra a resposta antes de finalizar. Adicionar como passo nos briefs de workflow que envolvem integração de APIs, ou nota em `.claude/rules/api-design.md` / `frontend.md`.
Ficheiro alvo: `.claude/rules/api-design.md` (secção nova "Parsing de APIs de terceiros: verificar contra resposta real").

## 3. Workflow paralelo recriou um componente em vez de o partilhar

| Campo | Valor |
|-------|-------|
| **Categoria** | `workflow-gap` |
| **Severidade** | `medium` |
| **Componente** | `.claude/skills/frontend.md` (produção/workflow) |

**Problema:** No workflow de build, agentes paralelos construíram a página de filme e a de série. O agente do filme usou o componente partilhado `VideoPlayer`; o da série criou um `<video>` inline próprio (controlos nativos, sem o OSD de comando). Resultou em dois players inconsistentes — apanhado pelo `design-review` e corrigido depois (unificação manual).

**Root cause:** Agentes paralelos por página não têm visibilidade do que os outros constroem, por isso cada um resolve "preciso de um player" localmente. Faltou definir os componentes partilhados numa fase de fundação ANTES do fan-out.

**Fix sugerido:** Em workflows de build/design com agentes paralelos por página/feature, definir os componentes partilhados (player, card, etc.) numa fase de fundação sequencial primeiro, e instruir os agentes de fan-out a IMPORTAR esses componentes, nunca recriá-los. Adicionar como princípio à secção de produção/workflow do `frontend.md`.
Ficheiro alvo: `.claude/skills/frontend.md` (secção "Production stack / workflow" — nota sobre componentes partilhados antes do fan-out).
